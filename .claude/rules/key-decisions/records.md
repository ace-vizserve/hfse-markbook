### KD #66
Master class template tables (migrations 031+032). Sections + subject weights configuration is driven by two master tables (`template_sections` + `template_subject_configs`) decoupled from any specific AY, instead of being copy-forwarded from the most-recent prior AY at AY-setup time. Superadmin manages the master at `/sis/admin/template`; the Apply panel propagates to one or many target AYs via `apply_template_to_ay(p_ay_slug, p_overwrite)` RPC — non-destructive UPSERT preserves operational data on already-populated AYs. Migration 032 seeds the master from AY9999. AY-setup wizard reads from the template tables when creating a new AY (replacing the pattern that pulled test-AY pollution into AY2026/AY2027). Migration 030 ships the idempotency upgrade for `create_academic_year` (safe to re-run on partial-state AYs; source-AY copy-forward filters out test AYs via `^AY9`). KD #4 + KD #40 stay valid — the template is the *source* the per-AY rows are *applied from*, not a replacement for the per-AY storage.

### KD #67
Atomic mid-year section transfer. Reassigning an enrolled student to a new section within the same level happens through the dedicated `POST /api/sis/students/[enroleeNumber]/transfer-section` route, NOT the existing class-stage PATCH. Helper `lib/sis/section-transfer.ts::transferStudentSection` runs an atomic withdraw-old-row + insert-new-row pair against `section_students` (per Hard Rule #6: append-only) plus an admissions-side `classSection`/`classLevel`/`classStatus='Finished'` update on the same call. The class-stage PATCH rejects post-Enrolled `classSection` mutations with a 422 pointing to the new endpoint — closes the silent dual-section bug where the per-row sync's narrow scope left the student active in BOTH sections. Pre-flight rejections: cross-level transfer (same-level only), target at 50-cap (Hard Rule #5), same-section, missing studentNumber. Term-of-transfer captured in audit context via shared `lib/sis/terms.ts::getTermForDate`. UI: per-row "Move" action on the SIS Admin section roster table at `/sis/sections/[id]`. Records detail page renders a "Section transfers" history card from `lib/sis/section-history.ts` (audit-log-derived, cross-AY).

### KD #68
Late-enrollee term detection. When the registrar tags a student as `late_enrollee` via `<EnrolmentEditSheet>`, the section-students PATCH refreshes `enrollment_date=today` on the active→late_enrollee boundary (boundary-only — idempotent re-saves don't re-stamp the date) so the joining-term lookup against `terms` reflects the registrar's tag moment, not the row's original sync-creation date. Term resolved via shared `lib/sis/terms.ts::getTermForDate(date, ayCode)` helper which also powers the section-transfer route (KD #67). Audit context captures `lateEnrolleeTermLabel`. Records placement section's StatusBadge gets an inline "· T2" amber suffix for late-enrollee rows, derived from a preloaded terms-by-AY map (`preloadTermsForAYs` + `termForDateInPreloaded`). Falls back to "between terms" when the date sits outside any defined term window.

### KD #72

### KD #83
Enrolment movements page is audit-log-derived, not table-backed (Sprint 36). `/records/movements` (registrar + school_admin + superadmin) consolidates three event kinds — section transfer, withdrawal, late-enrol — by reading `audit_log` directly: `student.section.transfer` rows for transfers; `enrolment.metadata.update` rows demuxed by `context.after.enrollment_status === 'withdrawn'` (withdrawn) and `context.lateEnrolleeTransition === true` (late-enrol). No new table, no new mutation surface — the existing transfer + roster-edit routes are the canonical writers, the page is a passive aggregation reader. **Withdrawal lumps "transferred to another school"** (no destination tracking — the withdrawn enum is the single signal). **Default scope = current AY**; `?scope=all` toggle via `<Switch>` (newly-installed shadcn primitive at `components/ui/switch.tsx`) widens to all AYs. **Counts are derived from the same events array** so the KPI-strip cards always match the table (resolves the previous design's scope-mismatch + PostgREST 1000-row truncation in one move). Loader at `lib/sis/movements.ts` exports a discriminated `MovementEvent` union; transfer rows resolve studentName/Number via the AY-apps table while metadata rows resolve via `section_students.id` UUID → joined section + level + AY → `students` table. Term enrichment via `preloadTermsForAYs` once per fetch (KD #68 pattern). Linkified student name routes to `/records/students/[studentNumber]` with `/records/students/by-enrolee/[enroleeNumber]` fallback per KD #81. **Seeder shim** at `lib/sis/seeder/movements.ts` writes ~12 synthetic audit rows per test AY (5 transfers in T1 / 4 withdrawals spread T1–T2 / 3 late-enrols in T2) using actor_email `'registrar.seed@hfse.test'`; idempotent via the existing transfer-row count check; audit-only writes (does not mutate section_students rosters since that would cascade through grades/attendance/evaluation). **Sprint 37 bugfix**: `seedEnrolledAdmissionsRows` now back-writes the deterministic `enroleeNumber` onto `section_students.enrolee_number` (migration 041 added the column but the seeder was never wired — every row had it as NULL, so seedMovements's transfer-eligibility filter (`!!s.enroleeNumber`) returned zero rows and the function short-circuited before withdrawals + late-enrols could be inserted either). `seedMovements` no longer short-circuits when `transferEligible` is empty.

### KD #85
Section roster re-alphabetize via Postgres RPC (Sprint 37, migration 042). New `realphabetize_section_index_numbers(p_section_id uuid)` re-assigns `section_students.index_number` 1..N alphabetically by `(last_name, first_name, middle_name)`. Active + late-enrollee rows come first; withdrawn rows append at the bottom in the same order. **Two-phase update via negative-index staging** — the `unique (section_id, index_number)` constraint is non-deferrable so a naive `UPDATE SET index_number = N WHERE id = X` loop hits violations on the first iteration; the RPC instead flips every row to its negative value (still unique, no conflicts) then assigns positives via `row_number() OVER (...)` in a CTE-driven UPDATE. Returns `{ rows_renumbered, before, after }` JSONB so the audit log captures the full rename map. Surfaces via `<RealphabetizeRosterButton>` on `/sis/sections/[id]` (registrar + school_admin + superadmin); confirm dialog explains that grades/attendance/evaluations are NOT affected (they reference section_students by UUID, not by index_number). Audit-logged as `section.realphabetize`; invalidates all operational drill caches for the AY post-update. **Pattern reusable**: any future mutation surface that needs to safely permute a column under a non-deferrable unique constraint should reuse this negative-index staging — don't reach for `DEFERRABLE INITIALLY DEFERRED` retrospectively, and don't try a JS-loop sequential UPDATE.

### KD #90
Unsynced enrolled students — graceful records lite page + queue + assign-section flow. Closes the chronic Directus drift where admissions rows land in Enrolled status without `classSection` set, so `syncOneStudent` skips with `'missing classLevel or classSection'` and `public.students` never gets the row. **Before**: `/records/students/[studentNumber]` redirected to admissions when `findStudentByNumber` returned null. **After**: the page falls through to a `<RecordsLitePage>` RSC rendering identity from admissions history, an amber `<UnsyncedActionCard>` (primary "Assign to a section" button + outline "Open in admissions" link), and empty states across the non-Overview tabs. `notFound()` only fires when admissions history is also empty. **New `POST /api/sis/students/[enroleeNumber]/assign-section`** handles two cases: standard assign (Step A admissions UPDATE classSection/classLevel/classStatus='Finished' → Step B `syncOneStudent` → Step C audit `sis.student.assign_section` → Step D revalidate; rolls back Step A if Step B fails) AND re-sync-only (classSection already set but `public.students` missing — skip Step A, run only the sync). **Operational queue at `/records/unsynced`** (registrar / school_admin / superadmin per KD #74): TanStack table per the unified `<DataTable>` shell (KD #84), per-row "Assign section" button + "Open in admissions" fallback, gap reason in plain English ("No class section assigned" / "No student number" / "Not yet synced"). **Sidebar entry** under Records → Operations with SSR-static count badge; **dashboard chip** on `/records` hero (amber, gated on count > 0 AND viewing current AY). Loader at `lib/sis/unsynced-students.ts` classifies each row by `gapReason`; cache tag `sis:${ayCode}` so existing mutation routes invalidate. Hard Rule #4 + KD #4 (cross-year via studentNumber) preserved — the lite page reads the existing `getEnrollmentHistory` data.

### KD #109
AY Readiness Indicator — `lib/sis/readiness.ts` + `<AyReadinessPill>` floating component in the SIS layout (`app/(sis)/layout.tsx`). Five ordered setup steps per AY: **1 AY Setup** (terms exist with dated start/end), **2 School Calendar** (school_day-type rows present for all terms), **3 Sections** (at least one section per level with a form adviser assigned), **4 Scheme of Work** (SOW published for all needed `(term × subject × level × curriculum_track)` scopes), **5 Grading Sheets** (all sections have grading sheets created). Each step returns a `ReadinessStep` with `status: 'done' | 'partial' | 'not_started'` and a `fraction?: { done, total }`. The pill is visible to `school_admin | superadmin` only (teachers and registrars do not see it). Dialog uses shadcn-native pattern (no external wrapper). **Pattern**: `getAyReadiness(ayCode)` is an `unstable_cache` wrapper; the SIS layout fetches it on every RSC render at `app/(sis)/layout.tsx` (service client, tag `sis:${ayCode}`). The SIS hub (`/sis`) redesigned with a "Year Setup" section showing 4 numbered `AdminCard`s (AY Setup, Calendar, Sections, SOW) and an Organisation section with a SOW card. The SIS sidebar gains a "Year Setup" nav group with 4 numbered step entries. **Note**: the `checkSections` step does NOT filter by `curriculum_track` — HFSE doesn't differentiate sections by track operationally; the column exists on `sections` and `template_sections` but the readiness check uses `(level_id, curriculum_track)` deduplication only for the SOW step's needed-scope calculation.
