## Evaluation — form-class-adviser writeups

### KD #49

Evaluation module owns the form-class-adviser write-up — sole source for T1–T3 report card FCA comments. `evaluation_writeups` is the source of truth, read by `lib/report-card/build-report-card.ts`. `terms.virtue_theme` drives the parenthetical: "Form Class Adviser's Comments (HFSE Virtues: …)". `/markbook/sections/[id]/comments` + `/markbook/grading/advisory/[id]/comments` redirect to Evaluation. T4 excluded (no FCA comment on the final card). `report_card_comments` is legacy (migration 018 backfilled into `evaluation_writeups`).

### KD #92

Per-topic 1–5 proficiency rating (migration 046). Replaces the binary `is_checked` UX on the Checklists tab with a 5-point qualitative scale: **1 Needs Improvement** (rarely demonstrates proficiency), **2 Developing** (occasionally), **3 Satisfactory** (sometimes), **4 Good** (frequently), **5 Excellent** (consistently demonstrates strong proficiency). Stored as `smallint` nullable with `CHECK (rating is null or rating between 1 and 5)`. `is_checked` retained for back-compat (no downstream consumers outside this module per the architect's audit; can be dropped later). **UI**: 5-button segmented control with word labels at `sm:` and above; degrades to `<Select>` ("Not rated" / "1 — Needs Improvement" / etc.) below `sm`. Active button uses tinted gradient per `RATINGS[i].swatchClassName` (destructive → amber → sky → indigo → mint, 1→5); inactive stays outline. Tooltip on each button shows the full description. Clicking an already-active button clears the rating. Constants in `lib/evaluation/ratings.ts` (single source: value + label + description + token classes). PTC use only — ratings do NOT flow to the report card (KD #49 unchanged). Auto-save fires per change; optimistic update with revert on PATCH failure. Component: `components/evaluation/rating-selector.tsx`. Progress badge reads "X / N rated" (counts `rating !== null`).

### KD #93

Teacher-owned evaluation topics — scope shift from `(subject × level × term)` to `(subject × section × term)` (migration 047). Reasoning: only the teacher who taught the class knows what they actually covered — curriculum-plan-as-written ≠ ground-truth-as-taught. **Schema**: drops `evaluation_checklist_items.level_id`, adds `section_id NOT NULL` references `sections(id) on delete cascade`, new index on `(term_id, subject_id, section_id, sort_order)`, new unique constraint on `(term_id, subject_id, section_id, item_text)` (blocks duplicate topics + makes copy-from idempotent via `onConflict ignoreDuplicates`). Hard reset on apply — existing admin-placeholder items deleted, cascades to responses. **Ownership shifts**: subject teachers (with `teacher_assignments.role='subject_teacher'` for the section+subject) own add / edit / delete / reorder. Registrar+ retain unrestricted write access. `/sis/admin/evaluation-checklists` converted from editable to **read-only audit view** — three-axis picker (term × subject × section), each row tagged with creator's display name + created-at date, amber banner explaining the new ownership model. `<ChecklistItemsEditor>` component deleted. **Teacher UI**: `<TopicManagerPanel>` inline above the student roster — "+ Add topic" inline input (Enter saves, Esc cancels), per-topic ▲ ▼ reorder buttons (two sequential PATCHes on sort_order; the unique constraint is on item_text, not sort_order), pencil-icon inline edit, trash-icon delete (with `<AlertDialog>` confirm noting that ratings cascade-delete). **"Copy topics from {section}"** shortcut surfaces when the current section's topic list is empty AND the teacher teaches the same subject in another section that has topics — new `POST /api/evaluation/checklist-items/copy-from` route uses `INSERT ... onConflict ignoreDuplicates` so the copy is idempotent. New AuditActions: `evaluation.checklist_item.reorder` + `.copy_from`. PATCH route distinguishes `update` from `reorder` based on whether item_text changed.

### KD #103

PTC feedback route + term resolver. `PATCH /api/evaluation/ptc-feedback` (registrar/school_admin/superadmin) records parent-teacher-conference feedback per `(student, term)`. Never flows to the report card (KD #49 unchanged) — PTC use only. `lib/evaluation/ptc-resolver.ts` derives which writeup term a PTC event "discusses" via pure date math against `calendar_events.category='ptc'` + `terms.end_date` — no hardcoded dates, no per-AY config. Rule: a PTC event discusses the most-recently-ended writeup term (T1/T2/T3 — T4 excluded per KD #49) whose `end_date` is before the PTC `start_date`. Consequence: the Apr PTC (which sits in T2) discusses T1 writeups; the Nov PTC (T4) discusses T3 writeups. This means the deadline-driving event for each writeup cycle is automatically derived from calendar data, so reschedules propagate without code changes. Multiple PTCs resolving to the same writeup term → earliest is the deadline driver.

### KD #106

Evaluation sections picker widened to subject-teacher assignments. `/evaluation/sections` was previously filtered to form-adviser-only sections, leaving subject teachers with no path to their sections. Two new helpers in `lib/evaluation/queries.ts`: `listSubjectTeacherSectionIds(userId)` queries `teacher_assignments` with `role='subject_teacher'`; `getChecklistTopicCountByTerm(termId, sectionIds)` counts `evaluation_checklist_items` per section. Picker fetches both sets in parallel and shows all sections where the teacher has any assignment. Card display is role-differentiated: adviser cards show write-up progress bar; subject-teacher-only cards show topic count + student count. Write-up stats strip hidden for pure subject teachers. Hero copy and empty state updated with role context.

### KD #107

~~Evaluation topics are admin-prescribed per `(subject × level × term)`.~~ **Superseded by KD #110.** The pending Chandana clarification was never conclusively answered; instead, a deeper workflow investigation revealed that the SOW itself (including WW/PT labels and evaluation topics) is teacher-owned, not coordinator-prescribed. KD #93's per-section scope was actually correct — it was KD #107's "admin-prescribed" framing that was wrong. Migration 061 (KD #110) re-scopes `evaluation_checklist_items` back to `section_id` and re-enables teacher-owned topic management via the SOW editor at `/markbook/sow`.

### KD #114

Evaluation module purpose — FCA write-ups only (Sprint 47, 2026-05-30). Field investigation (Chandana) confirmed PTC uses Ms. JoAnn's offline form; the SIS is not part of that process. The speculative PTC features built in KD #92, #93, #103, #106 (checklist topics, 1–5 ratings, subject comments, conference notes tab, PTC deadline banners) were removed from the UI and API. **What was removed:** Topics/Checklists tab, Conference Notes tab, all PTC banners from the roster page, topic count from the sections picker, subject-teacher access to the module. **API routes 410'd:** `POST /api/evaluation/checklist-items`, `PATCH/DELETE /api/evaluation/checklist-items/[id]`, `PATCH /api/evaluation/checklist-responses`, `PATCH /api/evaluation/subject-comments`, `PATCH /api/evaluation/ptc-feedback`. **Files deleted:** `lib/evaluation/checklist.ts`, `lib/schemas/evaluation-checklist.ts`, `components/evaluation/{checklist-roster-client,ptc-roster-client,rating-selector,checklist-subject-picker}.tsx`. **DB tables kept dormant** (`evaluation_checklist_items`, `evaluation_checklist_responses`, `evaluation_subject_comments`, `evaluation_ptc_feedback`) — schema intact for optional future PTC surface. `lib/evaluation/ptc-resolver.ts` kept — the evaluation dashboard uses it for PTC date awareness. **KD #106 reversed:** subject teachers no longer access `/evaluation`; only form advisers (write-ups) and registrar+ (read-only oversight). **Future PTC surface:** when HFSE is ready, build as a completely separate route/tab with zero coupling to the write-up flow — DB tables are ready.
